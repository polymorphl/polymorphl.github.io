Whether leading teams or more recently working with agents, I've always run into the same problem: enforcing architectural decisions. Not code style, not naming conventions. The real decisions. The ones that were discussed, validated, documented. And that get silently violated three weeks later, without bad intent, just from a lack of time to walk every developer through them individually.

For a long time, the partial answer to this problem came down to one word: the linter.

[ESLint](https://eslint.org/), [Prettier](https://prettier.io/), then [Biome](https://biomejs.dev/). Tools that plug into pre-commit hooks, CI, and editors. Fast, automatable, impossible to bypass without an explicit `--no-verify`. The linter won that battle. And it deserves to.

But there's a limit. A limit that existed long before agents, and that AI has simply made impossible to ignore.

## What the linter can do

On a well-maintained NestJS API, a linter covers the basics without effort:

- Code style: quotes, semicolons, trailing commas, line endings
- Forbidden imports: barrel exports enforced via `noRestrictedImports`, impossible to import from `orders/domain/` instead of `orders/`
- TypeScript conventions: `interface` banned in favor of `type`, `any` forbidden outside explicitly commented cases

These rules share one thing: they're **mechanical**. They express themselves in syntax. The linter can verify them without knowing anything about the project, its intentions, its business constraints. That's precisely its strength.

And its limit.

## What the linter can't see

Some decisions can't be expressed in syntax. They express themselves in **intent**.

Take a concrete case. We decided that controllers never touch the database directly. Everything goes through a service. The rule is clear, the team is aligned, it's in the wiki.

ESLint can't express that. It sees two valid TypeScript imports. It doesn't know that one of them betrays a fundamental architectural decision.

Another example: we decided to never throw a native `Error` in a service. We use NestJS exceptions so the global filter intercepts them correctly. A `throw new Error('Order not found')` is syntactically perfect. The linter sees nothing wrong. Yet it's a violation.

Same with DTOs: every incoming body must be typed with `class-validator`. A `@Body() body: any` passes the linter without a murmur.

The linter protects form. Not meaning.

<Callout type="warning">
This gap existed before agents. They've made it impossible to ignore: an agent can generate in seconds as much code as an entire sprint, at a speed that human review can't absorb.
</Callout>

## ADRs, and why they stay passive by default

ADRs (*Architecture Decision Records*) aren't new. [Michael Nygard described the format in 2011](https://cognitect.com/blog/2011/11/15/documenting-architecture-decisions.html): one Markdown file per decision, with three sections. The context that made the decision necessary. The decision itself. The consequences, positive and negative.

The format is simple, the principle is sound. But the classic problem with ADRs: they're **passive**. A Markdown file the team is supposed to re-read before contributing. In practice, it drifts. New joiners don't read all of them. PR reviews cover too much code to cross-reference every change with every decision. The seniors who wrote them end up being the only ones who remember them. And agents don't read them at all, unless explicitly asked.

## From passive documentation to active constraint

That's where the paradigm shifts, and not just for agents. If ADRs are structured so their rules are extractable, any tool can understand them, apply them, and above all **verify** them. A pre-push hook, a CI step, a command run manually before a PR.

The key: adding an explicit section to each ADR that formalizes what's enforceable.

## `/adr-guard`: the behavior linter

The command is a Markdown file at `.claude/skills/adr-guard/SKILL.md`. Claude Code makes it available via `/adr-guard`.

Unlike what you might expect, it doesn't read the ADRs and deduce rules on the fly. It embeds a **checklist of concrete bash checks**, one per architectural decision. Mechanizable rules become `grep`s — an `@InjectRepository` in a controller, a `throw new Error` in a service. Rules that require context (verifying that a layer doesn't depend on another, validating a state lifecycle) remain manual checkpoints that the agent reviews explicitly. The final report consolidates both.

<Callout type="tip">
The linter and `/adr-guard` don't do the same job. Together, they close the gap.
</Callout>

### Three levels of integration

The command adapts to any workflow, human or agentic.

**Level 1 — Manual in Claude Code**

You run `/adr-guard` after an implementation session, before submitting a PR. The report appears in the terminal, you fix before pushing.

<Callout type="info">
The report is also directly exportable as Markdown: `claude -p "/adr-guard" --output-format text > report.md` — handy for attaching to a PR or archiving a compliance snapshot.
</Callout>

**Level 2 — Automatic hook with Lefthook**

[Lefthook](https://lefthook.dev/) installs as a dev dependency and hooks into git. `lefthook.yml` is enough to run `/adr-guard` automatically on every `git push`:

```yaml
# lefthook.yml
pre-push:
  commands:
    adr-guard:
      run: |
        report=$(claude -p "/adr-guard" --output-format text)
        echo "$report"
        ! grep -q "✗" <<< "$report"
```

`claude -p` always exits with code 0 — it's the report content that determines the block. The wrapper captures the output, prints it, then fails if it contains violations (`✗`). Violations detected, push blocked, report in the terminal. Same reflex as Biome on style, different register.

**Level 3 — CI**

Same logic as a GitHub Actions step to block merges to main:

```yaml
- name: ADR Guard
  run: |
    report=$(claude -p "/adr-guard" --output-format text)
    echo "$report"
    ! grep -q "✗" <<< "$report"
```

A human dev, an agent, a PR in CI: the circuit is always the same. ADRs define the rules, `/adr-guard` verifies them, the report shows what to fix.

## What actually changes

The role of architectural arbiter doesn't disappear with `/adr-guard`. The model reads ADRs more or less well depending on its configuration, some violations slip through, and there are still decisions that require human judgment that no `grep` can replace.

But it shifts the cursor. A portion of the PR review that served only to verify documented conventions can be automated. That recovered time goes to what actually matters: new decisions, edge cases, tradeoffs that don't have an ADR yet.

ADRs were always right on substance. The problem wasn't the format, it was the absence of real enforcement. A Markdown file with no tool to enforce it is an honor system. `/adr-guard` makes them enforceable. Lefthook makes them automatic. CI makes them unavoidable.

The linter took a decade to become a reflex, from JSHint to Biome. Active ADRs are at the beginning of that path.
